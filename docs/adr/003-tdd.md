# ADR-003: Red-Green TDD on the Domain Layer

**Status:** Accepted

## Context

Test discipline pays off in long-term maintainability — but only when it's targeted at code that benefits. Android tests (Espresso, instrumented Compose) are slow, brittle, and require an emulator or device. Pure-Kotlin tests on the JVM are fast (sub-second), deterministic, and pleasant.

## Decision

**Red-Green TDD on every domain function. No tests on shell layers.**

### Domain (`domain/`)

- Every exported function has a failing test BEFORE implementation
- Tests live in `src/test/kotlin/` (JVM unit tests, no Android)
- Test file is colocated: `Snooze.kt` → `SnoozeTest.kt`
- One concept per test file
- Use JUnit 5 + Kotlin's `kotlin.test` assertions

### Shell (`data/`, `ui/`, `notification/`)

- **Not unit-tested.** Verified via on-device manual testing for each phase's verification steps.
- If shell code is doing enough that it feels worth testing, that's a signal to lift the branching logic into `domain/` and test it there.

## Workflow

1. Identify a new domain function to add (e.g., `applySnooze(session, now): SnoozeSession`)
2. Write a failing test describing the expected behavior — multiple cases as separate `@Test` methods
3. Run tests; confirm failure with a meaningful diff
4. Write the minimum implementation to pass
5. Run tests; confirm green
6. Refactor if needed; tests stay green
7. Commit

Conventional commit format per ADR-006.

## Rationale

- **Domain code is where bugs are most expensive.** A bug in the snooze math affects every notification refresh; a bug in a Compose layout is visible and easy to fix.
- **Domain code is also the cheapest to test.** Pure functions, no setup, sub-second runtime.
- **Shell code is glue.** Wrong glue is visible at runtime; tests of glue mostly verify the glue is wired up, which on-device testing does for free.

## Consequences

- The domain layer becomes the natural home for any logic that's worth getting right — which reinforces the architectural separation (ADR-002).
- We have to be disciplined about not letting branching logic accumulate in ViewModels or services. Code review catches this.
- Test coverage metrics will look "low" because shell code isn't covered. That's fine — coverage as a metric is misleading; what matters is that the testable code is tested.

## Anti-patterns

- Tests that mock half the world to test a ViewModel — sign that domain logic leaked upward
- Snapshot tests of Compose UI — high churn, low value, breaks every time you adjust spacing
- Integration tests that boot the Android runtime to verify domain math — domain should be JVM-testable, period
